# MEMORY_POLICY.md

**记忆工厂的政策定义** —— 什么可以入库，什么必须丢弃，什么只能当状态。

---

## 一、什么是"长期记忆"

长期记忆是**提炼后的稳定领域知识**，必须满足：
- 经过真实代码/文档/构建过程验证
- 经过用户确认（human-in-the-loop approve）
- 具有较长生命周期，不会随日常开发频繁变更
- 服务于四类场景之一：新人入项 / 三方应用 / 新需求开发 / 问题定位修复

---

## 二、什么是"状态"

状态是**探索过程中的临时工作数据**，不得入库，只能写 STATE.md：
- 当前探索到哪了
- 哪些证据缺口已识别但尚未确认
- 下一轮计划探索什么
- 正在审批中的草案
- 已被拒绝但待重新探索的草案

---

## 三、什么是"证据"

证据是**支撑记忆的原始材料**，独立存放在 EVIDENCE/ 目录：
- 代码片段（附路径 + 符号锚点）
- 文档摘录（附文档路径）
- 构建/运行日志
- commit 关联
- 用户确认记录（含飞书卡片 ID + 时间戳）

证据可以很多，不得直接写入 MEMORY/。

---

## 四、什么必须丢弃

以下内容**永远不得**作为长期记忆入库：
- 未经证实的架构假设
- "应该是..."类陈述（without evidence）
- 全量代码 dump
- 全文索引
- 临时的调试注释
- 非 AVCodec / DFX 领域的知识

---

## 五、入库必须通过的 8 条门禁

每条记忆在写入 Git 之前必须通过：

| # | 规则 | 说明 |
|---|------|------|
| 1 | 有 summary | 简洁的总结，能让读者快速理解 |
| 2 | 有 evidence | 至少一条有效 evidence |
| 3 | 标注了 scope | 明确属于哪个子领域/模块 |
| 4 | 区分 stable knowledge / temporary state | 稳定知识才能标 stable |
| 5 | 关联四类场景之一 | 必须有明确的服务场景 |
| 6 | 无猜测性语言 | 禁止 "应该是 / 可能是 / 大概是" |
| 7 | 有 owner | 谁负责维护这条记忆 |
| 8 | 有 update trigger | 标注什么情况下需要重新验证 |

---

## 六、记忆分层定义

| 层 | 名称 | 说明 |
|----|------|------|
| L1 | Identity / Constitution | Agent 身份、规则、行为准则（本文件 + AGENT_CONSTITUTION.md）|
| L2 | Long-term durable memory | 稳定知识：架构/边界/术语/流程/工具链/FAQ |
| L3 | Working state | 当前探索状态、待确认、文件扫描记录、下轮计划 |
| L4 | Evidence index | 所有知识的证据索引 |

---

## 七、Git 发布规则

- 所有 stable 记忆必须经过 approve 才能写入 Git
- Commit message 格式固定：`[MEM] <topic> - <short summary>`
- 每次发布必须同步更新 CHANGELOG.md
- 禁止强制覆盖主分支，必须通过 PR 或受控分支机制

---

## 八、变更再学习规则

当 AVCodec 代码发生变更时：
1. 识别受影响 memory entry
2. 标为 `stale`
3. 进入 publish_queue.yaml
4. 触发重新探索相关主题
